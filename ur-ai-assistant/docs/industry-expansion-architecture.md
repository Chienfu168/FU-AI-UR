# UR AI Assistant 多產業擴充架構規劃

本文件記錄「以現有都更危老功能為基礎，逐步擴充為多產業共用平台」的架構規劃。
本階段只做規劃，**不改動任何現有程式碼與行為**，作為後續分階段實作的依循依據。

---

## 一、背景與目標

目前外掛的核心引擎（FAQ 知識庫、AI 問答、Quiz、行情參考、相關頁面推薦、熱門問題、
問答紀錄與回饋分析）本質上是「產業無關」的通用架構：它們處理的是「問答比對」、
「資料統計」、「內容推薦」這類與都更無關的通用問題。目前之所以整套外掛看起來是
「都更專用」，原因幾乎全部來自**內容與文案**（FAQ 題庫、AI 系統提示詞、前台標題、
行情參考的用語框架），而不是程式邏輯本身。

目標：讓同一套外掛，透過選擇「產業別」，就能切換 AI 回答方向、預設內容與模組組合，
服務以下四個產業別（本階段先規劃這四個，未來可再擴充）：

1. 都更產業（建設公司／規劃公司／都更顧問）—— 現有功能的基礎，作為預設產業別。
2. 不動產經紀業（房仲）
3. 地政士
4. 估價師

不採取「另開專案／分裝多套外掛」的做法，原因：核心引擎已高度可共用，分裝會讓同一個
bug 需要在多個 repo 各修一次，長期維護成本遠高於「單一引擎＋產業設定檔」。

## 二、現況模組盤點：哪些通用、哪些都更專屬

| 模組 | 現況定位 | 產業擴充時的處理方式 |
|---|---|---|
| 後台架構／AJAX／Logs／Feedback／相關頁面推薦／熱門問題 | 完全通用，與產業無關 | 不需調整 |
| FAQ 知識庫（比對／CRUD／後台管理引擎） | 引擎通用，**內容**（題庫 CSV）是都更專屬 | 引擎不動，題庫做成「產業內容包」 |
| Quiz 知識大考驗（出題／作答／排行榜引擎） | 引擎通用，**題庫內容**是都更專屬 | 同上，題庫做成「產業內容包」 |
| AI 助理核心（Answer Service） | **完全通用**：`system_prompt`、前台標題／副標題本來就是後台設定值，不是寫死在程式碼裡，只是目前的「預設值」是都更用語 | 只需要讓不同產業別提供不同的預設值 |
| 行情參考（實價登錄匯入／統計／自動抓取） | 引擎完全通用（實價登錄資料對房仲、估價師、地政士都是剛需），只有「都更效益排行榜」這個**命名框架**是都更專屬 | 抽出用語為可設定文案，保留統計邏輯不變 |
| 分回試算（權利變換計算機） | **真正的都更專屬業務邏輯**（換坪倍數、容積獎勵、地主分回比例，僅都更／危老權利變換情境適用） | 做成「依產業別預設開／關」的選用模組，房仲／地政士／估價師預設關閉 |

結論：真正需要「重寫邏輯」的只有分回試算一項；其餘都是「把寫死的預設值／文案抽成
可依產業切換的設定」，工程風險相對低。

## 三、「產業設定檔」（Industry Profile）機制設計

### 3.1 核心概念

新增一個全域設定 `industry`（產業別 key），存在既有的主設定（`UR_AI_Settings`）中，
預設值為 `urban_renewal`（都更危老）—— **確保所有現有安裝的行為完全不變**，因為預設
產業別就是目前唯一支援的都更危老。

新增一個「產業設定檔登錄檔」類別（暫定 `UR_AI_Industry_Profiles`），提供：

```
get_all()          // 所有產業別清單（key => 顯示名稱），供後台下拉選單使用
get(industry_key)  // 取得單一產業別的完整設定內容
get_active()        // 取得目前啟用中的產業別設定（讀取 industry 設定，找不到時退回 urban_renewal）
```

每個產業別的設定內容（資料，不是程式邏輯），初步規劃欄位：

```
key                       // 'urban_renewal' / 'real_estate_agent' / 'land_agent' / 'appraiser'
label                     // 後台顯示名稱
assistant.system_prompt   // AI 系統提示詞預設值
assistant.frontend_title  // 前台助理標題預設值
assistant.frontend_subtitle
modules.calculator        // 分回試算模組預設開／關
modules.market_price      // 行情參考模組預設開／關（四個產業別預期都開啟）
modules.quiz              // Quiz 模組預設開／關
market_price.ranking_label     // 排行榜命名（都更效益／區域行情比較…）
market_price.disclaimer_extra  // 各產業別可能需要的額外免責聲明（估價師尤其需要強調「不構成正式估價」）
content_packs.faq         // 建議匯入的 FAQ 起始包位置（見 3.3）
content_packs.quiz        // 建議匯入的 Quiz 起始包位置
```

### 3.2 與既有設定的關係：預設值，而非強制值

**很重要的原則**：產業設定檔只提供「預設值」與「一鍵匯入的起始內容包」，不取代管理者
既有的手動設定能力。例如：

- 系統提示詞、免責聲明，管理者今天本來就可以在設定頁自由編輯；有了產業別之後，只是
  「初次安裝或重置設定時，依所選產業別帶入不同的預設文字」，管理者仍可以隨時改寫。
- 分回試算模組是否啟用，沿用目前 `market_price`／`calculator` 各自既有的「功能啟用」
  設定（例如 `UR_AI_Market_Price_Settings::is_enabled()` 這種模式），只是這個布林值
  的**預設值**由產業別決定，管理者仍可手動開關。

這個設計確保：即使日後某個都更顧問想順便打開「房仲」的行情用語，或某個房仲網站想
額外開啟分回試算，都不會被架構綁死。

### 3.3 內容包（FAQ／Quiz 起始資料）

現況：目前的都更 FAQ（約 100 篇）與 Quiz 題庫（約 40 題）是過去在對話中產生後直接
匯入既有 WordPress 網站的資料，**沒有以檔案形式留在這個 repo 裡**。

規劃：把都更版的 FAQ／Quiz 內容整理成 CSV，存放到 repo 內固定位置（例如
`data/industry-packs/urban-renewal/faq.csv`、`quiz.csv`），作為「都更」產業別的起始包，
也作為未來其他產業別內容包的**格式範本**。後台可以新增一個「匯入產業起始包」的按鈕，
直接沿用現有的 FAQ／Quiz CSV 匯入機制（不需要新寫解析邏輯），只是資料來源從「使用者
上傳」變成「外掛內建檔案」。

這一步除了服務多產業擴充，也順便替現有都更內容做一次「版本化留存」，之後內容修訂
才有真正的變更歷史可查，而不是只存在於已匯入的資料庫與過去的對話紀錄裡。

### 3.4 品牌與命名（待決策，不在本階段處理）

外掛目前的名稱、後台選單標題、readme.txt 描述都是「都更危老 AI 助理」，這是產品定位
／行銷層級的決策，不是純工程問題。建議在完成第二個產業別的試點驗證、確認機制真的可行
之後，再一併決定：

- 是否需要把外掛對外名稱改為產業中性的名稱（例如「不動產 AI 助理」），都更危老變成
  其中一個產業別；
- 或是維持「都更危老 AI 助理」為主打名稱，其他產業別視為附加功能，不特別行銷。

本階段不預先決定，避免在機制都還沒驗證前就綁定品牌方向。

## 四、四個產業別的內容差異藍圖

以下先定義每個產業別「AI 回答方向」「FAQ 分類」「試算機／行情參考的角色」「Quiz 方向」，
作為之後實際寫內容時的依循框架，本階段不產出實際內容。

### 4.1 都更產業（建設公司／規劃公司／都更顧問）—— 基礎／預設產業別

- **AI 人設**：協助地主／一般民眾理解都市更新、危老重建、更新會、自主更新、權利變換、
  協議合建等法規與流程（沿用現有系統提示詞）。
- **FAQ 分類**：都市更新、危老重建、更新會、自主更新、權利變換、協議合建、容積獎勵、
  稅賦減免等（沿用現有 ~13 大類、約 100 篇）。
- **分回試算**：核心工具，預設開啟。
- **行情參考**：「都更效益排行榜」敘事（老屋現況 vs 新成屋行情落差），導向「這個社區
  都更後行情大概漲多少」。
- **Quiz**：都更／危老法規知識測驗（沿用現有 ~40 題）。

### 4.2 不動產經紀業（房仲）

- **AI 人設**：協助一般消費者理解買賣／租賃流程與消費者權益，強調合法仲介服務與資訊
  透明，語氣偏向「客戶教育」而非「法規講解」。
- **FAQ 分類（初步方向，待實際撰寫時再細化）**：委託銷售（專任／一般委託）、斡旋金與
  要約書、不動產說明書、價金信託／履約保證、產權調查基本概念、實價登錄申報義務、
  仲介服務費行情與計算方式、不動產經紀業管理條例相關消費者權益。
- **分回試算**：不適用，預設關閉。
- **行情參考**：核心工具之一，直接對客戶展示區域行情；用語從「都更效益」改為中性的
  「區域行情比較」或「新舊屋行情落差」。
- **Quiz 方向**：待確認優先順序——可作「不動產經紀人／營業員」考照複習用，也可作一般
  消費者的房地產知識互動遊戲，兩者內容取向不同，實作前需要再確認。

### 4.3 地政士

- **AI 人設**：協助民眾理解登記、過戶、稅務相關程序（輔助地政士向客戶說明流程，非地政士
  本人的專業考照工具）。
- **FAQ 分類（初步方向）**：所有權移轉登記、繼承登記、抵押權設定與塗銷、土地增值稅／
  契稅／印花稅、地籍測量基本概念、時效取得、預告登記、辦理登記所需文件清單。
- **分回試算**：不適用，預設關閉；地政士業務高度需要「稅費試算」（土地增值稅、契稅），
  這是最適合這個產業別的新試算模組雛形，但屬於未來擴充，不在本階段規劃範圍內。
- **行情參考**：次要工具，登記案件本身較不需要行情比較，但仍可作輔助參考保留。
- **Quiz 方向**：地政士考照重點複習。

### 4.4 估價師

- **AI 人設**：協助理解估價方法論、估價師法規、估價報告基本結構；**特別需要在系統提示詞
  與免責聲明中強調「僅供知識參考、不構成正式估價」**，避免使用者誤把 AI 回答當成正式
  估價意見，法遵風險相對其他產業別更高，需要特別小心處理。
- **FAQ 分類（初步方向）**：估價三大方法（比較法／成本法／收益法）、估價師法基本規範、
  估價報告基本知識、常見估價案件類型（金融擔保估價、法拍估價、都更權利變換估價等）。
- **分回試算**：不適用（估價師視角與都更實施者不同），預設關閉。
- **行情參考**：核心工具，甚至可以是這個產業別「主打」的模組；未來可再加深功能（例如
  更細緻的成交案例篩選條件），但本階段先沿用現有統計邏輯。
- **Quiz 方向**：估價師考照重點複習。

### 4.5 跨產業內容重疊的觀察

都更權利變換案件本身會牽涉到估價（權利變換估價），代表「都更產業」與「估價師產業」
在部分主題上有實質交集，未來寫內容包時可以互相參照、甚至共用部分文章，不需要完全
各自獨立撰寫，可以降低內容產出的總工作量。

## 五、分階段實作路線圖

- **Phase 0（本文件）**：定義產業設定檔資料結構、盤點現有都更專屬字串／內容、規劃各
  產業別的內容差異藍圖。**不改動任何程式碼。**
- **Phase 1（結構重構，不改變現有行為）**：
  1. 新增 `industry` 設定（預設值 `urban_renewal`）與 `UR_AI_Industry_Profiles` 登錄檔。
  2. 把現有寫死在 `UR_AI_Settings::default_system_prompt()` 與 `defaults()` 裡的都更
     文案，抽成 `urban_renewal` 產業設定檔的資料，讀取方式從「寫死字串」改為「讀取
     目前啟用產業別的預設值」——因為預設產業別就是 `urban_renewal`，這一步對所有現有
     安裝完全零影響，純粹是把「資料」從程式碼搬到設定檔結構。
  3. 行情參考的「都更效益排行榜」等文案抽成可設定字串，預設值不變。
  4. 分回試算模組的預設啟用狀態改為讀取產業設定檔（都更＝預設開，其餘暫時只有都更
     一個選項，尚不影響現況）。
  5. 把現有都更 FAQ／Quiz 內容整理成 CSV，存進 repo 作為 `urban_renewal` 內容包，
     新增「匯入產業起始包」的後台按鈕（沿用既有匯入機制）。
  6. 完整回歸測試：確認產業別只有 `urban_renewal` 一個選項時，外掛所有前後台行為與
     升級前完全一致。
- **Phase 2（試點第二產業）**：選定一個產業別實際撰寫 FAQ／Quiz 內容包、行情參考
  文案、系統提示詞，驗證整套「選擇產業別→切換方向」機制在真實內容下是否成立。
  **已完成，試點產業別為地政士**（而非原規劃建議的不動產經紀業，依實際商業判斷
  調整）：
  - 新增 `land_agent` 產業設定檔：AI 系統提示詞（強調地政登記／稅務僅供一般性
    參考，不可替代個案核算與正式送件確認）、前台標題／副標題、分回試算模組
    預設關閉、行情參考排行榜文案（「雙北區域行情漲幅排行榜」／「行情漲幅」欄位）。
  - FAQ 起始包（16 篇，`data/industry-packs/land_agent/faq.csv`）涵蓋所有權移轉
    登記、繼承登記、抵押權設定與塗銷、土地增值稅、契稅與印花稅、地籍測量、
    時效取得、預告登記、地政士角色說明。
  - Quiz 起始包（12 題，`data/industry-packs/land_agent/quiz.csv`），難度分布
    easy／medium／hard。
  - 兩份內容包皆以既有（未修改）的 `parse_faq_csv()`／`parse_quiz_csv()` 私有
    方法實際解析驗證過，確認能透過現有的「上傳 CSV」表單直接匯入，不需要
    額外的「一鍵匯入產業起始包」按鈕（此為後續可再評估的便利性優化，非本輪
    必要項目）。
  - 後台設定頁新增「產業別」下拉選單＋「套用此產業別的預設文案」按鈕（純前端
    預覽填入，不會自動儲存，避免誤蓋既有客製化內容）。
  - 驗證：撰寫回歸測試腳本，確認（a）未設定產業別時輸出與升級前逐字元相同；
    （b）切換為地政士後，AI 人設、前台標題、分回試算預設關閉、行情榜單文案
    皆正確切換；（c）無效產業別 key 會安全退回都更危老預設值，不會出錯。
- **Phase 3／4**：依序補齊地政士、估價師的內容包，內容產出可參考 4.5 的跨產業重疊
  觀察降低工作量。
- **待決策事項**：外掛對外品牌命名（見 3.4），建議在 Phase 2 驗證完成後再決定。

### 5.1 都更範圍細分：廣義都更／危老 vs. 自主更新（追加）

實際使用後發現，即使同屬「都更／危老」大類，「由建商或都更顧問主導」與
「地主自組更新會、自主推動」在受眾與溝通方向上仍有明顯差異，因此把原本
單一的 `urban_renewal` 產業別拆成兩個：

- `urban_renewal`：廣義都更重建／都市更新（含危老重建；建設公司／規劃公司
  ／都更顧問），沿用原本的系統提示詞與內容，行為與拆分前完全一致。
- `self_renewal`：狹義、僅限「自主更新」，系統提示詞刻意限縮在地主自組
  更新會、不透過建商主導的範圍，與廣義都更分開溝通。

兩者都新增 `promotion` 欄位（`site_label`／`site_url`），對應站方自營的
兩個網站，在前台以低調的「本服務由 OO 提供」附連結方式曝光（見
`UR_AI_Industry_Profiles::render_promotion_attribution()`）；沒有配置
`promotion` 的產業別（例如地政士）前台不會顯示任何內容。這個欄位刻意設計
成「有就顯示、沒有就不顯示」，未來新增產業別時不強制要求要有對應網站。

## 六、風險與工作量評估摘要

- **工程風險**：低。多數改動是「把寫死字串搬到設定檔」，不涉及資料庫 schema 變更、
  不影響既有使用者資料，且每一步都可以先在只有 `urban_renewal` 一個產業別的狀態下
  完整測試，確保零回歸。分回試算模組的「依產業別預設開關」是唯一涉及模組啟用邏輯的
  改動，但沿用既有「功能啟用」設定模式，風險同樣可控。
- **真正的成本**：**內容產出**。每個新產業別大約需要對等於都更版的內容量（約 100 篇
  FAQ、40 題測驗），這是五個目標產業裡最花時間的部分，遠大於架構調整本身的工程量。
- **建議**：Phase 1 结構重構風險低、可以連續執行；Phase 2 起的內容產出建議一次做一個
  產業別、實際驗證後再往下一個推進，避免一次投入大量內容產出後才發現機制需要調整。
